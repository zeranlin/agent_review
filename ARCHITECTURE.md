# 架构说明

## 目标

构建一个可复现、可复核、可审计的政府采购招标文件合规审查 harness。  
这个 harness 不依赖一次性 prompt 做“拍脑袋判断”，而是把 `sop.md` 要求的审查流程拆成稳定的层级结构、显式工件和固定输出。

系统目标包括：

- 让智能体按统一 SOP 执行审查
- 让每个问题都能回落到具体条款和具体证据
- 让人工审查员能够接手复核，不依赖上下文记忆
- 让后续规则、OCR、法规知识库和多智能体协作都能平滑扩展

## 为什么采用这种形态

本仓库借鉴了 OpenAI 强调的两个思路：

- 在 [Harness engineering](https://openai.com/zh-Hans-CN/index/harness-engineering/) 中，仓库本身就是记录系统，规则、流程、约束和输出结构应沉淀在代码与文档中。
- 在 [Unrolling the Codex agent loop](https://openai.com/zh-Hans-CN/index/unrolling-the-codex-agent-loop/) 中，核心循环是：目标输入 -> 工具调用 -> 观察结果 -> 更新计划 -> 最终输出。

把这两个思路映射到招标文件合规审查场景后，意味着：

- 先定义审查边界，再开始判断
- 先抽取事实条款，再做规则匹配
- 先识别限制竞争等高风险，再看模板和规范性问题
- 先形成结构化证据，再输出审查意见

## 审查主循环

`agent_review` 的主循环严格对应 `sop.md`：

1. 接收采购文件及可选附件。
2. 解析单个或多个关联文件，抽取文本、页码、段落、表格和附件引用。
3. 识别文件类型，并生成审查范围与审查边界说明。
4. 定位关键章节，形成章节位置索引表。
5. 抽取关键条款，形成条款抽取表。
6. 对抽取结果运行风险规则匹配，形成风险命中表。
7. 对跨条款信息执行一致性检查，形成一致性检查表。
8. 综合风险等级、证据完整性和审查边界，生成总体结论。
9. 输出主要问题、相对规范项和逐条修改建议。
10. 持久化整套运行产物，供人工复核和后续 agent 复用。

在多文件场景下，正文、采购需求附件、评分细则、合同草案、补遗文件应先形成“联合审查上下文”，再进入后续抽取、规则和一致性阶段。

联合审查上下文之上，还应显式运行“跨文件一致性专项”，至少覆盖：

- 正文 vs 评分细则
- 正文 vs 合同草案
- 后续可扩展到正文 vs 补遗/澄清文件

## 慢 LLM 前提下的执行策略

本项目默认接受一个现实前提：LLM 响应可能显著慢于规则链路。

因此架构上必须遵循：

- 确定性主链路先完成
- LLM 作为增强层，而不是阻塞型主依赖
- 所有 LLM 任务都必须支持超时、降级和跳过
- 中间工件必须落盘，避免重复调用 LLM

## 双模式运行

### 1. `fast` 模式

只执行确定性链路：

- 文件解析
- 文件类型识别
- 章节定位
- 条款抽取
- 风险匹配
- 一致性检查
- 基础结论生成
- 基础报告输出

适用场景：

- 批量审查
- 首轮筛查
- 对时延敏感的场景

### 2. `enhanced` 模式

先执行完整 `fast` 链路，再进行 LLM 增强：

- 条款抽取后的语义补全
- 专项规则后的语义复核
- 一致性矩阵后的深层冲突分析
- 高风险结论前的裁决复核
- 总体结论润色与修改建议增强

适用场景：

- 正式审查意见
- 需要更自然表述的报告
- 人工复核前的增强版本输出

## 系统分层

### 1. 接口层

负责“如何发起审查、如何消费结果”。

包含：

- CLI 命令入口
- 批量任务入口
- Markdown 输出
- JSON 输出
- 后续可扩展的 Web/API 接口

职责：

- 接收输入文件路径或任务参数
- 选择输出格式
- 将运行结果暴露给人工或下游系统

### 2. 编排层

负责“按什么顺序执行 SOP”。

包含：

- 运行参数校验
- 审查任务初始化
- 按 stage pipeline 分阶段调用解析、抽取、规则匹配和报告生成
- 中间工件汇总与写出
- 运行索引写出

职责：

- 严格按照 SOP 顺序推进
- 控制各层的输入输出边界
- 在必要时触发人工复核升级
- 在 `enhanced` 模式下先产出基础报告，再调用 LLM 增强
- 在 LLM 超时或失败时自动降级回基础报告
- 通过 `stage_records` 和 `run_manifest.json` 记录每次执行

### 3. 输入与解析层

负责“原始文件如何变成可用文本”。

支持对象：

- `pdf`
- `docx`
- `doc`
- 图片扫描件

能力包括：

- 文件存在性校验
- 文件格式识别
- PDF 页码和文本抽取
- DOCX 段落、表格、标题抽取
- 扫描件 OCR
- 图片预处理与多轮 OCR
- 图片表格的结构化抽取
- 视觉 OCR 增强识别（图片类型、摘要、关键字段）
- 附件与表格的引用保留

产物包括：

- 文件基础信息表
- 原始文本提取结果
- 表格提取结果

### 4. 文档结构层

负责“文件是什么、可以审到哪里、关键章节在哪里”。

包含：

- 文件类型识别
- 审查范围说明生成
- 审查边界声明
- 章节定位与锚点建立

文件类型至少包括：

- 完整招标文件
- 采购需求文件
- 评分细则文件
- 合同草案
- 混合型文件

职责：

- 判断是否可以对程序条款、评分条款、合同条款作完整定性
- 对不完整文件明确写出边界
- 为后续条款抽取提供稳定定位点

### 5. 条款抽取层

负责“应该审的事实条款有哪些”。

抽取范围包括：

- 项目基本信息
- 资格条款
- 技术条款
- 评分条款
- 合同条款
- 政策条款

每条抽取结果必须包含：

- 条款类别
- 字段名称
- 条款内容
- 来源定位

职责：

- 把文档中的事实转成结构化字段
- 为风险规则匹配提供标准化输入

### 6. 规则与风险层

负责“这些条款命中了哪些风险规则”。

重点风险域包括：

- 限制竞争风险
- 评分不规范风险
- 项目属性错配风险
- 合同与验收风险
- 模板残留风险
- 政策条款风险

职责：

- 对抽取结果执行规则匹配
- 通过规则注册中心统一管理通用规则和专项规则
- 采用“核心必跑 + 场景增强”的双层规则架构
- 输出风险命中名称、理由、严重程度、来源定位
- 区分高风险、中风险和低风险/待完善

其中：

- 核心必跑规则始终执行，不依赖场景命中
- 场景增强规则根据项目属性、文件内容和少量场景标签做增强
- 场景识别只影响增强深度，不作为是否审查的硬门槛

### 7. 一致性与裁决层

负责“前后条款是否矛盾、结论如何分级”。

一致性检查至少包括：

- 项目属性 vs 履约内容
- 项目属性 vs 合同类型
- 技术要求 vs 评分标准
- 技术要求 vs 验收标准
- 预算金额 vs 最高限价
- 中小企业政策 vs 价格扣除政策
- 质保要求 vs 合同质保条款
- 采购需求 vs 合同草案用语
- 联合体/分包条款前后一致性

裁决输出至少包括：

- 风险等级表
- 总体结论
- 人工复核清单

职责：

- 对跨条款冲突问题单列为重点问题
- 根据风险数量、风险级别和证据完整性给出结论层级
- 对无法核实的问题升级人工

### 8. 证据与报告层

负责“如何形成可复核的最终产物”。

标准输出必须包括：

- 审查范围说明
- 总体结论
- 主要问题
- 相对规范项
- 修改建议

正式版输出可继续扩展：

- 审查依据
- 条款定位
- 风险等级
- 审查意见书格式

职责：

- 将中间工件整合成结构化报告
- 保证一条问题对应一条修改建议
- 为高频风险和一致性问题挂接结构化法规依据
- 对 LLM 语义补充结果做“可直接采用 / 需人工确认”分级
- 将高风险问题和待人工确认问题单独沉淀为人工复核产物
- 让后续人工或 agent 可以继续复用这些产物
- 同时输出基础报告和增强报告两套产物
- 同时输出专项表和运行索引文件

## 编排细化

当前编排层已经显式拆成以下 stage：

1. `document_structure`
2. `clause_extraction`
3. `dimension_review`
4. `rule_evaluation`
5. `consistency_review`
6. `finalize_report`

这样可以把 `engine.py` 从厚流程脚本收敛为轻量编排入口，并让每一步都保留稳定的中间状态、计数信息和执行记录。

## 运行产物

每次审查运行至少应产出以下结构化工件：

- 文件基础信息表
- 原始文本提取结果
- 表格提取结果
- 文件类型标签
- 审查范围说明
- 审查边界说明
- 章节位置索引表
- 条款抽取表
- 风险命中表
- 一致性检查表
- 风险等级表
- 总体结论
- 主要问题清单
- 相对规范项清单
- 问题-建议对应表
- 人工复核清单

在慢 LLM 前提下，还应额外产出：

- `base_report.json`
- `base_report.md`
- `enhanced_report.json`
- `enhanced_report.md`
- `run_manifest.json`
- `llm_tasks.json`
- `high_risk_review_checklist.json`
- `pending_confirmation_items.json`
- 五类专项表独立 JSON
- stage 执行记录
- LLM warning / timeout 信息

其中 `llm_tasks.json` 用于显式记录 4 个 LLM 语义子任务：

1. `llm_clause_supplement`
2. `llm_specialist_review`
3. `llm_consistency_review`
4. `llm_verdict_review`

每个子任务都应具备独立状态，至少包括：

- `pending`
- `running`
- `completed`
- `failed`
- `timed_out`
- `skipped`

对于 LLM 语义补充结果，还应增加结果分级：

- `rule_based`：来自规则链和确定性主链路的结果
- `可直接采用`：LLM 补充结果置信度较高，可直接合并到增强报告
- `需人工确认`：LLM 补充结果存在歧义或上下文依赖，需要进入待确认问题单

## 设计原则

- 证据优先：所有问题必须落到具体条款和锚点。
- SOP 优先：固定流程先于自由发挥，避免漏审高风险环节。
- 确定性基线：核心抽取、匹配和分级逻辑应尽量可重复。
- 审查边界明确：不因为材料不全而越界定性。
- 高风险优先：先看限制竞争、评分、合同和属性错配，再看模板问题。
- 原子能力和业务判断分离：OCR、解析、规则、裁决和报告渲染保持解耦。
- 对 agent 友好：输出工件必须便于下一轮 agent 继续接力。
- 可扩展：未来可以无缝接入法规知识库、OCR 增强和多智能体协作。
- LLM 可降级：任何增强失败都不能阻断基础审查结果。

## 建议目录映射

如果继续扩展，推荐目录逐步演进为：

```text
agent_review/
  ARCHITECTURE.md
  AGENTS.md
  docs/
    business_design.md
    dimension_design.md
    review_workflow.md
  src/agent_review/
    cli.py
    engine.py
    models.py
    pipeline.py
    merge.py
    reporting.py
    parsers/          # 输入与解析层
    structure/        # 文件类型识别、章节定位、范围声明
    extractors/       # 条款抽取层
    rules/            # 风险规则、注册中心与分级
    consistency/      # 一致性检查与裁决
    outputs/          # 标准化报告与工件写出
```
